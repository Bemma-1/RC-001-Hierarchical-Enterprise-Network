## 1. Purpose

This Low-Level Design defines the detailed technical implementation of the RC-001 enterprise network.

It specifies:

* Devices
* Hostnames
* Interface assignments
* VLANs
* IPv4 subnets
* Default gateways
* WAN addressing
* Trunk links
* Access ports
* OSPF parameters
* SSH settings
* Management addressing
* Server addressing
* Validation requirements

The Packet Tracer topology should be built from this document.

## 21. SSH Management Standard

All routers and switches shall use:

* Unique hostname
* Domain name
* RSA keys
* SSH Version 2
* Local administrative username
* VTY authentication using local credentials
* SSH-only VTY transport

Example domain:

`algl.local`

Published configuration files shall not expose real personal passwords.

Portfolio-safe placeholder credentials may be used.

---

## 22. Basic Device Hardening

Where supported, devices shall include:

* `enable secret`
* Encrypted password handling
* `service password-encryption`
* MOTD banner
* SSH-only VTY lines
* Shutdown of unused ports
* Interface descriptions
* Disabled DNS lookup if useful for lab administration
* Console password or local login where appropriate

These measures provide baseline hardening only.

---

## 23. Interface Description Standard

Every infrastructure link should include a meaningful description.

Examples:

* `TO-HQ-SW1`
* `TO-ACC-R1`
* `TO-TAK-R1`
* `TO-HQ-R1`
* `SERVER-AD`
* `USER-HR`

This improves troubleshooting and configuration readability.

---

## 24. Cabling Plan

Recommended Packet Tracer cabling:

* Router-to-switch — Copper straight-through or automatic connection
* Switch-to-switch — Copper crossover or automatic connection
* Router-to-router Ethernet WAN simulation — Appropriate Ethernet connection supported by Packet Tracer
* End devices-to-switch — Copper straight-through

Packet Tracer automatic connection may be used where appropriate, but interface assignments must remain consistent with this LLD.

## 25. Logical Traffic Paths

### Headquarters User to Server

```text
PC
→ Access Switch
→ HQ-SW1
→ HQ-R1 subinterface routing if VLAN differs
→ HQ-SW1
→ Server
```

### Branch User to HQ Server

```text
Branch PC
→ Branch Switch
→ Branch Router
→ OSPF-routed WAN
→ HQ-R1
→ HQ Server VLAN
→ Server
```

## 28. Implementation Sequence

The recommended build sequence is:

1. Place routers and switches.
2. Rename infrastructure devices.
3. Cable all links.
4. Create VLANs.
5. Configure switch access ports.
6. Configure trunks.
7. Configure router subinterfaces.
8. Configure WAN IP addresses.
9. Configure switch management SVIs.
10. Configure endpoints.
11. Configure OSPF.
12. Configure SSH.
13. Apply basic hardening.
14. Test local VLAN connectivity.
15. Test inter-VLAN routing.
16. Test OSPF.
17. Test inter-site communication.
18. Test server reachability.
19. Test SSH.
20. Save configurations.
21. Capture screenshots.
22. Run experiments.
23. Record deviations and lessons learned.

---

## 29. Known Design Limitations

The baseline implementation does not include:

* Redundant routers
* Redundant WAN links
* Layer 3 switching
* ACL-based departmental policy
* Enterprise firewalling
* VPN encryption
* AAA
* IDS/IPS
* IPv6
* Wireless access

These limitations are intentional and create clear opportunities for future projects.
