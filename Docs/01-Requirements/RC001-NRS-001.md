## 1. Purpose

This Network Requirements Specification defines the business, functional, technical, security, operational, and scalability requirements for the design and implementation of a secure hierarchical enterprise network for Ashanti Logistics Ghana Ltd.

The document establishes the requirements baseline for:

* RC001-HLD-001 — High-Level Design
* RC001-LLD-001 — Low-Level Design
* Network implementation in Cisco Packet Tracer
* Testing and validation
* Experimental evaluation
* Final project documentation

All major design and implementation decisions should be traceable to one or more requirements in this document.

---

## 2. Business Background

Ashanti Logistics Ghana Ltd. is a fictional Ghanaian logistics and supply-chain organization used as the enterprise case study for RC-001.

The organization operates from:

* Headquarters — Kumasi
* Branch Office — Accra
* Branch Office — Takoradi

The organization provides services including:

* Freight forwarding
* Fleet and transport operations
* Warehouse coordination
* Customs and logistics support
* Supply-chain administration
* Internal corporate services

Its business processes depend increasingly on reliable access to shared applications, centralized data, internal communications, and networked services.

The organization therefore requires a structured enterprise network capable of supporting current operations while providing a foundation for future expansion.

---

## 3. Business Problem

The assumed legacy environment lacks a standardized enterprise architecture.

Key limitations include:

* Flat or insufficiently segmented LAN design
* Excessive broadcast-domain size
* Limited separation of departmental traffic
* Manual and difficult-to-scale routing
* Insecure or inconsistent device-management practices
* Lack of a dedicated network-management segment
* Poor scalability for additional users and sites
* Limited documentation and configuration standardization
* Difficulty troubleshooting faults across multiple locations

The proposed network must address these limitations while remaining practical for implementation within Cisco Packet Tracer.

---

## 4. Project Objectives

The project shall:

1. Design a structured multi-site enterprise network.
2. Connect Kumasi headquarters with Accra and Takoradi branches.
3. Implement a hierarchical LAN architecture at headquarters.
4. Divide major organizational functions into separate VLANs.
5. Implement Layer 3 routing between required networks.
6. Implement OSPF for dynamic enterprise routing.
7. Provide centralized server connectivity.
8. Implement encrypted device administration using SSH Version 2.
9. Create a dedicated network-management VLAN.
10. Establish standardized device naming and IP addressing.
11. Demonstrate scalability by using a modular addressing and routing design.
12. Validate the design through structured tests and experiments.
13. Produce professional engineering and research-oriented documentation.

---

## 5. Project Scope

### 5.1 In Scope

RC-001 includes:

* Kumasi headquarters LAN
* Accra branch LAN
* Takoradi branch LAN
* WAN links between headquarters and branches
* Hierarchical switching at headquarters
* VLAN segmentation
* IEEE 802.1Q trunking
* Router-on-a-Stick inter-VLAN routing where appropriate
* IPv4 addressing
* OSPF dynamic routing
* Centralized server network
* Dedicated management VLANs
* SSH Version 2 device administration
* Basic Cisco IOS hardening
* Network testing
* Design documentation
* Research-style experiments

### 5.2 Out of Scope

The following are intentionally deferred to later projects:

* Enterprise firewall deployment
* Access Control Lists as a comprehensive departmental security policy
* Site-to-site VPN encryption
* SD-WAN
* Wireless LAN design
* IPv6
* VoIP
* Network Access Control
* TACACS+ or RADIUS
* SIEM integration
* Intrusion detection or prevention
* Cloud integration
* Full high-availability architecture
* Redundant WAN links

These exclusions keep RC-001 focused on secure network architecture fundamentals and establish a baseline for more advanced projects.

* Human Resources

* Management network
* WAN connectivity to headquarters

### 6.3 Takoradi Branch

The Takoradi branch shall contain:

* One branch router
* One managed access switch
* User network
* Management network
* WAN connectivity to headquarters

## 8. Functional Requirements

### FR-01 — Multi-Site Connectivity

The network shall provide Layer 3 connectivity between:

* Kumasi headquarters
* Accra branch
* Takoradi branch

### FR-02 — Departmental Segmentation

Headquarters departments shall be placed in separate VLANs and IP subnets.

### FR-03 — Inter-VLAN Routing

The network shall support Layer 3 communication between required VLANs.

Because comprehensive ACL enforcement is outside RC-001 scope, inter-VLAN routing shall primarily demonstrate connectivity and segmentation architecture rather than strict departmental authorization policy.

### FR-04 — Centralized Services

Users at headquarters and branches shall be capable of reaching centralized server resources hosted at headquarters.

### FR-05 — Dynamic Routing

The enterprise WAN shall use OSPF to dynamically advertise internal networks.

### FR-06 — Secure Administration

Routers and switches shall support SSH Version 2 for encrypted remote administration.

### FR-07 — Management Segmentation

Network infrastructure shall use dedicated management VLANs and management IP addresses where supported.

### FR-08 — Standardized Naming

All infrastructure devices shall use a standardized hostname convention based on site and device function.

### FR-09 — Scalable Addressing

The IP addressing plan shall provide sufficient capacity for current users and reasonable future growth.

### FR-10 — Branch Expansion

The architecture shall support the addition of future branches without requiring a complete network redesign.

---

## 9. Non-Functional Requirements

### NFR-01 — Scalability

The architecture shall use modular addressing, dynamic routing, and repeatable branch design patterns.

### NFR-02 — Maintainability

Configurations shall use consistent:

* Hostnames
* VLAN IDs
* Interface descriptions
* Addressing conventions
* Documentation standards

### NFR-03 — Availability

The network shall provide stable operation under normal conditions.

Full redundancy is outside RC-001 scope and will be addressed in RC-002.

### NFR-04 — Performance

Broadcast domains shall be reduced through VLAN segmentation.

WAN addressing shall use appropriately sized point-to-point subnets.

### NFR-05 — Security

Administrative traffic shall use encrypted SSH rather than Telnet.

Network management shall be logically separated from normal user traffic.

Unused access ports should be administratively disabled where practical.

### NFR-06 — Usability

The network design shall be understandable, reproducible, and suitable for troubleshooting by a trained network administrator.

### NFR-07 — Documentation

The implemented environment shall be documented through:

* NRS
* HLD
* LLD
* Architecture Decision Records
* IP plan
* Risk register
* Test plan
* Configuration files
* Screenshots
* Experimental results
* Research journal

## 10. Security Requirements

### SR-01

Telnet shall not be used for normal remote administration.

### SR-02

SSH Version 2 shall be enabled on managed Cisco infrastructure.

### SR-03

Local administrative credentials shall be protected using supported Cisco IOS password-protection mechanisms.

### SR-04

Network-device management traffic shall use dedicated management VLANs where applicable.

### SR-05

Departmental networks shall use separate VLANs and subnets.

### SR-06

Unused switch access ports should be shut down where practical.

### SR-07

Device banners shall identify systems as authorized-access-only infrastructure.

### SR-08

Configurations shall avoid exposing plaintext administrative passwords in published documentation.

### SR-09

More advanced policy enforcement, including ACLs, firewalls, AAA, VPNs, and Zero Trust controls, shall be treated as future enhancements rather than falsely represented as part of RC-001.

---

## 11. Routing Requirements

### RR-01

OSPF shall be used as the internal dynamic routing protocol.

### RR-02

RC-001 shall use a single-area OSPF design using Area 0.

### RR-03

Branch LAN networks shall be advertised into OSPF.

### RR-04

Headquarters VLAN networks shall be advertised into OSPF.

### RR-05

WAN point-to-point networks shall participate in OSPF.

### RR-06

The design shall permit future evolution toward multi-area OSPF if the enterprise grows significantly.

`<SITE>-<DEVICE-TYPE><NUMBER>`

* HQ-R1
* HQ-SW1
* ACC-R1
* ACC-SW1
* TAK-R1
* TAK-SW1

Server naming shall identify service role.

* DNS-SRV
* FILE-SRV

## 16. Testing Requirements

The completed implementation shall validate:

* VLAN creation
* Access-port assignment
* Trunk establishment
* Inter-VLAN routing
* OSPF neighbor establishment
* OSPF route learning
* Headquarters-to-branch connectivity
* Branch-to-headquarters connectivity
* Server reachability
* Management connectivity
* SSH login functionality
* Route recovery following a simulated WAN interruption

---

## 17. Experimental Requirements

The project shall include at least four structured experiments.

### EXP-01 — OSPF Convergence

Investigate routing recovery after simulated WAN link failure.

### EXP-02 — VLAN Segmentation

Evaluate how VLANs divide broadcast domains compared with a flat LAN.

### EXP-03 — Secure Administration

Compare the security characteristics of SSH-based administration with insecure plaintext remote-management approaches conceptually or through supported simulation.

### EXP-04 — Scalability

Evaluate how easily the addressing and routing design can accommodate an additional branch.

---

## 18. Acceptance Criteria

RC-001 shall be considered successfully implemented when:

1. All required infrastructure devices are configured.
2. Required VLANs exist and are correctly assigned.
3. Trunks operate correctly.
4. Inter-VLAN routing functions.
5. OSPF neighbors form successfully.
6. All intended internal routes appear in routing tables.
7. Branch users can reach authorized centralized infrastructure required by the baseline design.
8. SSH management works.
9. No Telnet dependency is required.
10. Testing evidence is captured.
11. The implemented topology matches the LLD or documented deviations are recorded.
12. Experimental observations and conclusions are published.

---

## 19. Assumptions

The project assumes:

* WAN transport between locations is available.
* Cisco Packet Tracer provides sufficient feature emulation.
* Qualified administrators operate the infrastructure.
* Physical security of network equipment is handled separately.
* Internet connectivity is simulated or conceptual unless explicitly implemented.
* RC-001 emphasizes architecture, routing, segmentation, and secure administration rather than complete enterprise cybersecurity enforcement.

---

## 20. Constraints and Limitations

* Packet Tracer is a simulation environment.
* Real-world throughput and latency measurements cannot be treated as production-equivalent benchmarks.
* Device features may differ from physical IOS/IOS XE platforms.
* High availability is not fully implemented.
* VLANs alone do not provide complete access-control isolation.
* Advanced cybersecurity controls are intentionally deferred.
