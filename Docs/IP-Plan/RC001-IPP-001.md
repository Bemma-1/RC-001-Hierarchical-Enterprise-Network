# 1. Purpose

This document defines the IPv4 addressing architecture for RC-001 — Secure Hierarchical Enterprise Network.

The addressing plan provides a structured allocation of IPv4 address space for:

* Kumasi Headquarters
* Accra Branch
* Takoradi Branch
* Departmental VLANs
* Server infrastructure
* Network management
* WAN point-to-point links
* End-user devices
* Future expansion

The plan is designed to support scalability, operational clarity, efficient subnet utilization, and simplified troubleshooting.

This document supplements RC001-LLD-001 and serves as the authoritative addressing reference during implementation and validation.

---

# 2. Addressing Design Principles

The addressing architecture follows six principles.

## 2.1 Site-Based Addressing

Each physical site receives a dedicated private IPv4 address block.

This allows administrators to identify the approximate location of a device by examining its IP address.

## 2.2 VLAN-to-Subnet Mapping

Each VLAN is assigned a unique IP subnet.

A VLAN and its corresponding IP subnet represent a single Layer 2 broadcast domain.

## 2.3 Variable Length Subnet Masking

VLSM is used to allocate subnet sizes according to estimated host requirements rather than assigning identical subnet sizes to every department.

## 2.4 Gateway Standardization

Where practical, the first usable address in each subnet is assigned to the default gateway.

Example:

`10.10.10.1`

This creates a predictable addressing convention.

## 2.5 Infrastructure Reservation

Low-numbered addresses are reserved for gateways, switches, servers, and other infrastructure.

## 2.6 Future Growth

Large site-level address blocks are reserved so additional VLANs and services can be introduced without redesigning the enterprise addressing architecture.

# 3. Enterprise Addressing Hierarchy

The following private IPv4 blocks are reserved:

| Function/Site       | Address Block | Purpose                                |
| ------------------- | ------------- | -------------------------------------- |
| Kumasi Headquarters | 10.10.0.0/16  | HQ departments, servers and management |
| Accra Branch        | 10.20.0.0/16  | Accra users and infrastructure         |
| Takoradi Branch     | 10.30.0.0/16  | Takoradi users and infrastructure      |
| WAN Infrastructure  | 10.255.0.0/16 | Routed point-to-point links            |

This structure creates an easily recognizable hierarchy:

`10.10.x.x` → Kumasi Headquarters

`10.20.x.x` → Accra

`10.30.x.x` → Takoradi

`10.255.x.x` → WAN infrastructure

Large portions of each `/16` remain intentionally unallocated for future growth.

---

# 4. VLSM Design Methodology

Subnet sizes are selected according to estimated host requirements.

The general calculation is:

`2^h - 2 ≥ required hosts`

where:

* `h` = number of host bits
* `-2` accounts for the network and broadcast addresses

The resulting subnet must provide enough usable addresses for current requirements plus reasonable growth.

# 11. WAN Addressing

Point-to-point WAN links use `/30` subnets.

A `/30` provides:

* 4 total addresses
* 2 usable host addresses
* 1 network address
* 1 broadcast address

This is suitable for the simulated point-to-point IPv4 links used in RC-001.

# 13. Address Allocation Convention

The following convention shall be used where practical.

| Address Range             | Intended Use                   |
| ------------------------- | ------------------------------ |
| First usable address      | Default gateway                |
| Low addresses             | Infrastructure                 |
| `.10` onward              | Representative endpoints/users |
| Static reserved addresses | Servers and managed devices    |

This convention improves consistency and troubleshooting.

---

# 14. Static vs Dynamic Addressing

## Static Addresses

Static addresses shall be used for:

* Router interfaces
* Router subinterfaces
* Switch management interfaces
* Servers
* Infrastructure devices

## User Endpoints

Representative Packet Tracer endpoints may initially use static addresses to make validation reproducible.

DHCP is not required for the baseline RC-001 implementation.

Future versions may introduce centralized DHCP and DHCP relay.

---

# 15. Addressing Scalability

Only small portions of the site `/16` blocks are allocated.

For example:

`10.10.0.0/16`

provides a large address space for future headquarters expansion.

Potential future networks might include:

* Voice VLAN
* Wireless employee VLAN
* Guest wireless VLAN
* Security systems VLAN
* IoT VLAN
* Additional server networks
* Network monitoring
* Backup infrastructure
* New departments

These additions can be accommodated without renumbering existing networks.

---

# 16. Future Branch Addressing Convention

Future branches should receive their own site blocks.

Example conceptual pattern:

* `10.40.0.0/16` — Future Branch 1
* `10.50.0.0/16` — Future Branch 2

This maintains geographic identification and simplifies routing policy.

Actual allocations shall be documented before implementation.

---

# 17. Address Conflict Prevention

Before assigning a new subnet or static IP address, the administrator shall verify:

1. The address is not already allocated.
2. The subnet does not overlap an existing subnet.
3. The address is not the network address.
4. The address is not the broadcast address.
5. The correct subnet mask is used.
6. The correct gateway is configured.
7. The assignment is documented.

---

# 18. Validation Requirements

The addressing plan shall be validated using:

* `show ip interface brief`
* `show ip route`
* `show ip route ospf`
* `show vlan brief`
* `show interfaces trunk`
* `ping`
* `traceroute` or `tracert` where supported

Tests shall confirm:

* Correct subnet masks
* Correct gateways
* No duplicate IP addresses
* No overlapping subnets
* Local VLAN connectivity
* Inter-VLAN connectivity
* WAN connectivity
* OSPF route propagation
* Server reachability
* Management reachability

# 19. Addressing Risks

## Incorrect Subnet Mask

Impact:

Loss of connectivity or unexpected local/remote host behavior.

Mitigation:

Validate all subnet masks against this IPP.

## Duplicate Address

Impact:

Intermittent or failed connectivity.

Mitigation:

Maintain documented static assignments.

## Incorrect Default Gateway

Impact:

Device cannot communicate outside its subnet.

Mitigation:

Use the first usable address convention consistently.

## Subnet Overlap

Impact:

Routing ambiguity.

Mitigation:

Review all new allocations against the enterprise addressing hierarchy.
